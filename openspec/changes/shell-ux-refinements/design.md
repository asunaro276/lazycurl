## Context

現在の実装は次のようになっている。

- `internal/tui.App` が `modeShell`/`modeEditor` という2つの排他的な画面モードを持つ。`shell.Shell` の `Requests`/`Editor` パネルで `e` キーを押すと `OpenEditorMsg` が発火し、`App` が画面全体を `form.Editor`(全画面フォーム)に差し替える。他のパネル(Collections/Response/History/モードタブ/ステータスバー)はこの間まったく描画されない。
- `form.Editor` 自体は既に「フォーカスしたフィールドは即座にタイプ可能」というモデルを内部的に持っている(`focusName`/`focusMethod`/`focusURL`/`focusContent` のゾーンチェーンを `tab` で移動し、`focusContent` 内は `1`-`4` で Params/Headers/Auth/Body のサブタブを切り替え、`KVGrid` は独自の「セル編集ゲート」(`enter`で編集開始/`esc`でキャンセル)を持つ)。問題は「この編集可能な状態にたどり着くまでに `e` を押して別画面へ遷移する必要がある」ことにある。
- `shell.Shell.handleKey` は `q`/`s`/`[`/`]`/`1`-`4`/`tab`/`shift+tab` をパネル種別を問わず最優先で解釈してから、パネルごとのハンドラに委譲している。この構造のまま `form.Editor` を各パネルに埋め込むと、これらのグローバルキーがテキスト入力(URLの `[]` やクエリの数字、`s`を含む単語など)を奪ってしまう。
- ステータスバー(`viewStatusBar`)はエラー発生時に `s.statusMsg` をキーバインド一覧の代わりに表示するが、明確なクリアタイミングがなく、次にたまたま `statusMsg` を上書きする操作が起きるまで残り続ける。

## Goals / Non-Goals

**Goals:**
- G1: エラー/警告メッセージは表示から5秒後に自動でキーバインド一覧表示へ戻る
- G2: リクエスト編集中も、モードタブ・Collections・Response・History・ステータスバーが表示され続け、独立した全画面へ遷移しない
- G3: Requests(Collections)/Editor(Adhoc)パネルにフォーカスが移った後、追加のモード遷移キー(`e`)を押さずにフィールドの直接編集を開始できる
- G4: `?`ヘルプオーバーレイが編集中でも一貫して機能する
- G5: グローバルショートカットとフォーム内テキスト入力の衝突を安全に解消する

**Non-Goals:**
- `KVGrid` のセル編集ゲート(`enter`で編集開始/`esc`でキャンセル)など、フォーム内部の既存インタラクションの再設計
- 環境変数展開・curl実行・collectionのファイルフォーマットへの変更
- 未保存の変更を残したまま終了する際の警告ダイアログの追加(将来課題)
- Collectionsパネル(コレクション一覧そのもの)のインライン編集化。対象はリクエストの編集のみ

## Decisions

### D1. ステータスメッセージの自動クリア(世代トークン方式)

`Shell` に `statusMsg string` と `statusGen int` を持たせ、既存の `s.statusMsg = "..."` という直接代入をすべて `s.setStatus(msg string) tea.Cmd` ヘルパー経由に置き換える。`setStatus` は呼び出すたびに `statusGen` をインクリメントし、`tea.Tick(5*time.Second, func(time.Time) tea.Msg { return clearStatusMsg{gen: gen} })` を返す。`Shell.Update` は `clearStatusMsg` を受け取った際、`msg.gen == s.statusGen` の場合のみ `statusMsg` を空にする。

呼び出し元の多くは現状 `tea.Cmd` を返さない箇所(例: `handleCollectionsKey` 内の直接代入)にあるため、それらも `tea.Cmd` を返すように戻り値を伝播させる必要がある。

代替案として「無条件に5秒後クリア」も検討したが、連続してエラーが起きた場合に古いタイマーが新しいメッセージを消してしまう競合が発生するため、世代トークン方式を採用する。

### D2. 編集状態を `App` から `Shell` へ移す

`App` から `mode`(`modeEditor`)・`editor`・`editingCol`・`editingIndex`・`statusOverride` を削除する。`OpenEditorMsg` は廃止する。`form.Editor` のインスタンスは `Shell` が保持し、`Requests`/`Editor` パネルの描画・キー処理の中で直接使う。

`App` 自体は `colStore`/`envStore`/`shell` の配線役として残す(将来 Shell をまたぐ機能の受け皿になり得るため、`main.go` が `shell.Shell` を直接 `tea.Program` に渡す形への統合は今回は行わない)。

### D3. フォーカスチェーンの拡張(パネル間 Tab とフォーム内 Tab の統合)

Collections モードの `Requests` パネルに「リストゾーン」を導入する。パネルにフォーカスが移った直後はリストゾーンであり、これは現行の `handleRequestsKey`(`j`/`k`/`enter`/`n`/`c`/`d`/`x`/`E`)と全く同じ挙動を維持する。

リストゾーンで `tab` を押すと、選択中リクエストの埋め込みフォーム(`Name` → `Method` → `URL` → `content`)へフォーカスが移る。フォームの最後のゾーン(`content`)でさらに `tab` を押すと Shell レベルの次パネル(`Response`)へフォーカスが移る。`shift+tab` は逆順(`Name` で `shift+tab` を押すとリストゾーンへ、リストゾーンで `shift+tab` を押すと `Collections` パネルへ)。

Adhoc モードの `Editor` パネルにはリストゾーンがない(スクラッチリクエストは常に1件)。パネルにフォーカスが移った瞬間、直接フォームの最初のゾーン(`Name`)にフォーカスが当たる。`content` から `tab` で `Response` パネルへ、`Name` から `shift+tab` でモードタブ側へ抜ける。

フォーム内部の `FocusNext`/`SetTab` ロジックは `form.Editor` の既存実装をそのまま流用する。

### D4. グローバルショートカットとフォーム内テキスト入力の衝突回避

`Shell.handleKey` は、現在フォーカスがフォームゾーン(D3で定義したリストゾーン以外の範囲)にあるかどうかを最初に判定し、真であればグローバルショートカット(`q`/`s`/`[`/`]`/`1`-`4`)の解釈をスキップしてフォームへ直接ディスパッチする。`ctrl+c`(送信中キャンセル/終了)のみ常にグローバルとして扱う。

これに伴い、以下のキー割り当てを変更する(**BREAKING**):

- サブタブ切り替え(Params/Headers/Auth/Body): `1`-`4` → `[`/`]`。これにより `1`-`4` はフォームゾーン内では単なる数字入力として素通しでき、パネルジャンプ機能はフォームゾーンの外(リストゾーンや他パネル)でのみ有効なままにできる
- `[`/`]`: `content` ゾーンでは「サブタブ切り替え」、それ以外(リストゾーン・他パネル)では従来通り「Adhoc/Collectionsモード切り替え」
- 終了: フォームゾーン内では `q` は文字入力に使われるため、`ctrl+c` のみに一本化する。フォームゾーンの外では従来通り `q` でも終了できる
- Adhocの「コレクションへ保存」: 従来の `s` キーから `ctrl+s` に変更する(D5)
- リクエスト送信: 従来 `enter` で送信していたが、フォームゾーンでは `enter` は各フィールド(特にBodyタブでは改行)に使われるため送信には使えない。フォームゾーンでの送信キーを `ctrl+r`(Run)に一本化する。リストゾーン・他パネル(History等)では従来通り `enter` が機能する

リストゾーン(Collections専用)はテキスト入力を持たないため、これらのグローバルキーは従来通り機能する。

### D5. 保存(`ctrl+s`)とメモリ即時反映

フォームでのキー入力のたびに `form.Editor.ToRequest()` を呼び、`Shell` が保持する `requests[idx]`(Collections)または `adhocRequest`(Adhoc)へ即座に反映する。リストゾーンのサマリー行(Method/Name)もリアルタイムに更新される。

`ctrl+s` の意味はモードによって異なる:
- Collections: 現在の `requests` スライス全体を `colStore.SaveRequests` で `.http` ファイルへ書き込む(現行 `App.saveEditor` のCollections分岐と同等の処理を `Shell` 側に移す)
- Adhoc: 従来 `s` キーが起動していた `overlaySaveAdhoc`(保存先コレクション選択)を起動する

`ctrl+q`(編集破棄)は廃止する。入力は常時メモリへ反映されるため「破棄」という概念自体がなくなる。

### D6. ヘルプオーバーレイ

`?` キーの処理・`viewHelp()` は現行のまま維持する。ただし `?` はURLのクエリ文字列等で実際に入力されうる文字であるため、D4の「フォームゾーンはグローバルキーを迂回する」対象に含める。フォームゾーン内では `?` も文字として入力され、ヘルプを開くには一度 `shift+tab` 等でリストゾーン/他パネルへ戻る必要がある。

## Risks / Trade-offs

- [Collectionsモードでは「フォーカス直後にリストゾーン、`tab`を1回押してはじめてフィールド編集が始まる」という近似解になる。Adhocモードは完全に満たすが、Collectionsモードは「フォーカス後、最初のTabで即編集開始」に留まる] → 妥協点としてユーザーに提示済み。将来的にリストゾーンを廃止しさらに簡略化する余地は残す
- [サブタブキーを`1`-`4`から`[`/`]`へ、Adhocの保存キーを`s`から`ctrl+s`へ変更するのは既存ユーザーの手癖を壊す**BREAKING**な変更] → ヘルプオーバーレイとステータスバーのヒント文言を更新し、変更点を明示する
- [フォームゾーンでほとんどのグローバルキーを迂回させる実装は`handleKey`の分岐が複雑になり、「キーがどこにも届かず何も起きない」バグを生みやすい] → 各グローバルキーに対応する文字(`q`, `s`, `[`, `]`, 数字)がフォームゾーン内で実際にテキストとして入力されることを単体テストで明示的に検証する
- [ステータスメッセージの5秒タイマーが送信中に連続してエラーになった場合など、`tea.Tick`のクリアメッセージが積み残る可能性] → 世代トークンで無害化(D1)。比較のみのO(1)処理なのでパフォーマンス影響はない
- [`?`をフォームゾーンでは文字入力として扱うため、編集中に直接ヘルプを開く手段がなくなる] → リストゾーン/他パネルからはこれまで通り`?`が機能するため、致命的ではないと判断

## Migration Plan

外部への段階的ロールアウトは不要な単一モジュール内の変更。以下の順で実装する想定:

1. ステータスメッセージ自動クリア(D1) — 他の変更に依存しない独立した修正
2. `form.Editor` のサブタブキーを `1`-`4` から `[`/`]` へ変更(D3/D4の前提)
3. `Shell` にフォームゾーンとフォーカスチェーン拡張を実装(D2/D3)
4. グローバルキーのフォームゾーン迂回、`ctrl+s`/`ctrl+q` の付け替え(D4/D5)
5. `App.modeEditor`/`OpenEditorMsg` の削除(D2)
6. ヘルプオーバーレイ・ステータスバーの文言更新、`openspec/specs` の該当spec更新

データ永続化フォーマットへの変更はないため、ロールバックは通常の `git revert` で対応できる。
